== Generer une representation graphique du Knowledge Graph (graphify) via PlantUML

=== Ce que le plugin fait deja

Le pipeline actuel lit `graphify-out/graph.json` via `GraphifyPromptAdapter` et produit des **diagrammes de classes UML** par communaute :

1. `generateDiagramDocs` extrait les communautes du graphe (noeuds + edges)
2. `GraphifyPromptAdapter.buildPrompt()` genere un prompt structure : liste de classes + relations EXTRACTED uniquement
3. Le prompt est consomme par `processPlantumlPrompts` (LLM > `.puml` > PNG)

**Limite** : le resultat est un diagramme de classes UML statique par communaute. Ce n'est *pas* une representation du Knowledge Graph lui-meme — on perd la topologie globale, les arbetes croises entre communautes, les types d'arrete (EXTRACTED/INFERRED/AMBIGUOUS), les scores de confiance, les hyperedges, et les liens s�mantiques.

=== Ce que j'attends

Une **representation graphique du Knowledge Graph complet** generee par le plugin Gradle, notamment :

- Tous les noeuds du graphe (classes, concepts, fonctions, papiers, images…) comme sommets
- Les arretes avec leur **type visible** (EXTRACTED, INFERRED, AMBIGUOS) et leur **score de confiance**
- Les communautes comme clusters visuels (packages ou rectangles colores)
- Les liens s�mantiques croises entre communautes (les `semantically_similar_to` marques INFERRED)
- Les hyperedges (relations groupant 3+ noeuds)
- La capacite de filtrer par communaute, par type d'arrete, ou par intervalle de confiance

=== Approche preconisee

Ajouter une nouvelle tache Gradle (ex: `GenerateKnowledgeGraphDiagramTask`) qui :

1. **Parse `graphify-out/graph.json` directement** (pas de prompt LLM — les donnees sont deja la, pas besoin d'inf�rence)
2. **Genere du PlantUML natif** en utilisant les primitives de graphe :
   - `package` ou `rectangle` pour les communautes (clusters visuels)
   - `node` pour chaque sommet du graphe
   - `-->` pour les arretes EXTRACTED (trait plein, noir)
   - `..>` pour les arretes INFERRED (trait pointill�, avec label du score de confiance)
   - `--x` pour les arretes AMBIGUOS (trait rouge pointill�)
   - Couleurs par communaute (palette automatique)
3. **Propriete de filtre** via CLI :
   - `-Pplantuml.kg.community="service layer"` — une communaute specifique
   - `-Pplantuml.kg.edgeTypes="EXTRACTED,INFERRED"` — filtrer par type d'arrete
   - `-Pplantuml.kg.minConfidence=0.7` — seuil de confiance minimum pour les INFERRED
   - `-Pplantuml.kg.maxNodes=50` — limiter le nombre de noeuds (pour la lisibilite)
4. **Pas de LLM** — c'est une transformation deterministe JSON > PlantUML, rapide et reproductible
5. **Integration dans le pipeline** : la tache depend de l'existence de `graphify-out/graph.json` (comme `generateDiagramDocs`), et le `.puml` genere peut etre rendu en PNG par `plantumlService.generateImage()`

**Avantage** : on obtient en une commande Gradle ce que `graphify-out/graph.html` (vis.js) fait dans le navigateur, mais en sortie PlantUML — integrable dans la documentation AsciiDoc du projet, versionnable dans Git, et consultable hors ligne sans navigateur.
