=== EPIC 13 : Multi-provider API Pool — Rotation par tiers (Enterprise → Pro → Free)

**Statut** : 🟡 EN COURS
**Priorité** : 🔴 HAUTE
**Session** : 138-141

==== Contexte

Le pool actuel tourne en round-robin ou least-used, sans notion de coût ou de tier. Les stratégies `WEIGHTED` et `SMART` sont des stubs. `fallbackEnabled` est défini mais inutilisé. L'objectif est d'ajouter une rotation intelligente qui consomme les clés payantes en priorité (Enterprise puis Pro) et garde le freemium comme filet de sécurité.

**Principe** : Une clé `Enterprise` n'a pas de raison d'attendre qu'une clé `Free` soit épuisée. On descend dans les tiers au fur et à mesure que les quotas saturent.

**Ordre de consommation** : `ENTERPRISE` → `PRO` → `FREE`

==== Tâches

[cols="1,4,1"]
|===
| Priorité | Tâche | Statut

.2+^.^| 🔴 HIGH
| **ApiKeyEntry** : Ajouter `tier: KeyTier` (ENTERPRISE=3, PRO=2, FREE=1) + `weight: Double` pour raffinement intra-tier dans `apikey/models.kt`
| ⬜

| **TieredRotationStrategy** : Nouveau composant — trier les entrées par `tier` décroissant, dans un même tier utiliser le `weight` pour pondérer la sélection
| ⬜

.2+^.^| 🟡 MEDIUM
| **ApiKeyPool.getNextKey()** : Intégrer `TieredRotationStrategy` — quand le quota d'une entrée est saturé, passer à l'entrée ou au tier suivant (vers le bas)
| ⬜

| **fallbackEnabled** : Câbler dans la rotation — quand toutes les entrées Enterprise+Pro sont épuisées, basculer sur FREE (si enabled)
| ⬜

.2+^.^| 🟢 LOW
| **YAML schema** : Ajouter `tier` et `weight` dans `ApiKeyPoolEntry`, màj `plantuml-test-context.yml`
| ⬜

| **Cross-provider** : Un même pool peut mixer Gemini Pro + Mistral Free + Ollama (local), fallback inter-provider quand tous les tiers d'un provider sont épuisés
| ⬜
|===

==== Critères d'Acceptation

* [ ] `TieredRotationStrategy` sélectionne Enterprise en priorité, puis Pro, puis Free
* [ ] `ApiKeyPool.getNextKey()` descend les tiers quand quota saturé
* [ ] Fallback FREE fonctionne quand `fallbackEnabled=true`
* [ ] YAML : `pool[].tier` parsé et utilisé
* [ ] Tests unitaires : stratégie de rotation + fallback entre tiers (≥15 tests)
