=== 0. PUBLISH TO MAVEN LOCAL — OBLIGATOIRE APRÈS TOUTE MODIFICATION DU CODE SOURCE

_Toute modification du code source du plugin DOIT être suivie de `./gradlew -q publishToMavenLocal` AVANT tout test fonctionnel ou validation via un projet consommateur._

* ❌ _Interdit_ : Tester le plugin via le projet parent sans avoir republié le JAR
* ❌ _Interdit_ : Considérer que le code source modifié = le JAR consommé par le projet parent
* ✅ _Obligatoire_ : `./gradlew -q publishToMavenLocal` après chaque `compileKotlin` réussi si le code a changé
* ✅ _Obligatoire_ : `./gradlew -q publishToMavenLocal` avant `./gradlew generateKnowledgeGraphDiagram` depuis le projet parent

_Raison_ : Le projet parent (`plantuml-gradle/`) consomme le plugin via `mavenLocal`. Le code source corrigé ne prend effet QUE si le JAR est republié. Oublier cette étape = tester une version périmée = faux positifs.

=== 1. COMMITS/GIT

_L'agent NE DOIT JAMAIS_ exécuter de commit, push, merge, ou toute commande git modifiant l'historique _SANS permission explicite de l'utilisateur_.

* ✅ _Autorisé_ : `git status`, `git diff`, `git log`, `git show` (lecture seule)
* ❌ _Interdit_ : `git add`, `git commit`, `git push`, `git merge`, `git rebase` (sauf ordre explicite)

_Procédure obligatoire avant tout commit_ :
. Montrer les modifications (`git diff --stat`)
. Demander : "Veux-tu que je commit ces changements ?"
. _Attendre confirmation explicite_ ("oui", "commit", "vas-y")
. _Seulement après_ : exécuter le commit

=== 1b. FICHIERS DE CONFIGURATION — RÈGLE ABSOLUE DE SÛRETÉ

**PRINCIPE** : Tout fichier de configuration (YAML, JSON, XML, TOML, properties, .env) peut contenir des données sensibles non versionnées. Ne **jamais** mettre en danger l'intégrité du travail du développeur.

**INTERDICTIONS** :

* ❌ **NE JAMAIS écraser** un fichier config avec un `Write` complet quand un `Edit` partiel suffit
* ❌ **NE JAMAIS remplacer** des valeurs sensibles (tokens, mots de passe, clés API, URLs prod) par des valeurs factices
* ❌ **NE JAMAIS modifier** sans vérifier `git check-ignore` et `git ls-files` — un fichier non tracké n'a pas de `git restore`

**Procédure OBLIGATOIRE** avant modification :

. `git check-ignore <fichier>` + `git ls-files -- <fichier>`
. Si non tracké : **CRÉER UNE SAUVEGARDE** (`cp <fichier> <fichier>.bak`)
. **PRÉSERVER** les valeurs sensibles intactes
. Privilégier l'**edit partiel** sur l'écriture complète
. **Vérifier** après modification que les valeurs sensibles sont toujours là

_Leçon Session 2 (bakery-plugin)_ : Écraser un site.yml non versionné a détruit des tokens réels. Sans sauvegarde git (fichier ignoré), restauration impossible sans fallback externe. Le développeur aurait été bloqué définitivement.

=== 4. FICHIERS PERSONNELS — INTERDICTION DE MODIFIER

_L'agent NE DOIT JAMAIS_ modifier, écraser ou supprimer un fichier personnel de l'utilisateur (`~/.gitconfig`, `~/.bashrc`, `~/.ssh/config`, `~/.zshrc`, etc.) _SANS permission explicite de l'utilisateur_.

**INTERDICTIONS** :

* **NE JAMAIS** utiliser `Write` ou `Edit` sur un fichier `~/.*` ou `~/*` qui n'est pas dans le répertoire de travail du projet
* **NE JAMAIS** exécuter de commande touchant `~/.gitconfig`, `~/.bashrc`, `~/.ssh/*`, `~/.zshrc`, etc. même avec l'intention de "réparer" ou "configurer"
* **Toujours rester dans le working directory** du projet actuel
