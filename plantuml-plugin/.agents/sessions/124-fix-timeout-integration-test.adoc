== Contexte

Le test `ReindexPlantumlRagIntegrationTest` timeout systematiquement apres 1 minute.
Le test utilisait Testcontainers (PostgreSQL + pgvector) + le vrai modele ONNX
`AllMiniLmL6V2EmbeddingModel` dans un test unitaire avec timeout JUnit 30s / Gradle 60s.

== Diagnostics

=== Cause racine

**Triple latence cumulative** :

1. **Testcontainers** : Demarrage PostgreSQL container (~15-20s)
2. **AllMiniLmL6V2EmbeddingModel** : Modele ONNX ~5s par embedding
3. **3 embeddings par test** (diagram1 + diagram2 + history) Ă— 10 tests = ~150s total

Le timeout JUnit (30s) et Gradle (60s) ne pouvaient pas absorber cette latence.

=== Analyse architecturale

Le test utilisait `ProjectBuilder` (in-process, meme JVM) mais avec des dependances
au monde reel (Testcontainers + modele ONNX). Selon la regle d'or du projet :

* `ProjectBuilder` = test unitaire = memes deps que `src/test/` â†’ doit etre rapide (<30s)
* `GradleRunner` = test fonctionnel = `src/functionalTest/` â†’ peut etre plus long

== Lecons apprises

* **ProjectBuilder = meme JVM = meme classpath** : la reflexion `Class.forName()` fonctionne
  car le stub est dans `src/test/kotlin/` et le test aussi
* **Separation unitaire/fonctionnel** : un test avec `ProjectBuilder` doit mocker le monde reel
* **Latence cumulative** : Testcontainers (20s) + ONNX (5s/embedding) = timeout certain
  en dessous de 60s
* **Injection par system property** : plus simple que `@Internal` property Gradle
  pour les tests unitaires in-process
