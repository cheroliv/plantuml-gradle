== Contexte

Les tests echouaient depuis plusieurs sessions suite a un decalage entre le model string
utilise dans la config (`"gemini"`, `"claude"`) et l'enum `Provider` qui utilise
`GOOGLE` et `ANTHROPIC`.

== Diagnostics

=== Cause racine

**`Provider.valueOf(provider.uppercase())`** dans `LlmService.kt` ne supportait pas les alias :
- `"gemini"` → `Provider.valueOf("GEMINI")` → **IllegalArgumentException** (aucune valeur GEMINI)
- `"claude"` → `Provider.valueOf("CLAUDE")` → **IllegalArgumentException** (aucune valeur CLAUDE)
- `"unsupported-model"` → `Provider.valueOf("UNSUPPORTED-MODEL")` → **IllegalArgumentException**

L'enum `Provider` a aussi 10 valeurs (9 providers + UNKNOWN), mais le test attendait 9.

== Corrections

=== 1. LlmService.kt — Ajout de `resolveProvider()`

Fonction de mapping qui remplace `Provider.valueOf()` :

[source,kotlin]
----
private fun resolveProvider(name: String): Provider {
    val upper = name.uppercase()
    return when (upper) {
        "GEMINI" -> Provider.GOOGLE
        "CLAUDE" -> Provider.ANTHROPIC
        else -> Provider.entries.find { it.name == upper } ?: Provider.UNKNOWN
    }
}
----

- Ligne 84 : `toApiKeyEntry()` → `resolveProvider(this.provider)` au lieu de `Provider.valueOf(this.provider.uppercase())`
- Ligne 114 : `auditLogger.logInfo()` → `resolveProvider(provider)` au lieu de `Provider.valueOf(provider.uppercase())`

=== 2. ProviderEnumsTest.kt — Correction du count

- `should have 9 values` → `should have 10 values` (9 providers + UNKNOWN)
- Ajout test `Provider should contain UNKNOWN`
