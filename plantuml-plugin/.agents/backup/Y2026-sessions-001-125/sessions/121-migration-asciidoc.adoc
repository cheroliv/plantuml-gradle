= Session 121 — Migration AsciiDoc

**Date** : 22 avril 2026 | **Statut** : ✅ | **Type** : docs

== Contexte

Le PROMPT_REPRISE.md de la session 121 proposait une extraction vers bakery-gradle (Phase 2). L'utilisateur a refuse cette approche car elle etait architecturalement incoherente (un script consommateur ne doit pas etre une dependance d'un plugin). La session a ete reoriente vers la migration des fichiers Markdown vers AsciiDoc, decrite dans TODO.adoc.

== Actions

|===
| Fichier | Modification

| ~100 fichiers `.md` | Convertis en `.adoc` et supprimes
| `AGENTS.md` (nouveau) | Accroche minimale pour opencode (convention de nommage)
| `AGENT.adoc` (nouveau) | Directives completes en AsciiDoc (sans S)
| `opencode.json` (nouveau) | Configuration opencode pointant vers AGENT.adoc
| 90 fichiers `.adoc` | References croisees mises a jour (.md → .adoc)
|===

== Connaissances cles

* Opencode ne lit que `AGENTS.md` par convention — pas de support natif `.adoc`
* Solution : `AGENTS.md` minimal + `opencode.json` avec `"instructions": ["AGENT.adoc"]` pour double couverture
* Nomage : `AGENTS.md` (avec S, convention opencode) vs `AGENT.adoc` (sans S, choix utilisateur)
* 128 fichiers `.adoc` au total dans le repo apres migration
* 1 seul `.md` reste : `AGENTS.md`

== Lecons apprises

* L'extraction vers bakery-gradle etait une mauvaise idee : un script consommateur ne doit pas etre dependance d'un plugin
* Toujours valider la coherence architecturale avant de lancer un EPIC
* La convention de nommage opencode (`AGENTS.md`) est un constrainte reelle qui ne peut pas etre contournee nativement
* `opencode.json` + `instructions` est le mecanisme pour charger des fichiers `.adoc`
