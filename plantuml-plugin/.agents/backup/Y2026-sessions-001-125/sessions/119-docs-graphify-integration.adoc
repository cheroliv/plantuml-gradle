== 🔧 Connaissances clés

. **Graphify** — Plugin OpenCode (32K stars) qui transforme un dossier de code en knowledge graph queryable. Installation : `uv tool install graphifyy && graphify install --platform opencode`
. **Combo gagnant** — Stratégie gère le **quand/comment** (gouvernance), Graphify gère le **quoi/où** (structure). Les deux sont complémentaires, pas en compétition.
. **Réduction du EAGER** — ECOSYSTEM_OVERVIEW.adoc (225 lignes) + ARCHITECTURE.adoc (130 lignes) remplacés par GRAPH_REPORT.adoc (~50 lignes). Économie : -36% sur le EAGER, -93% sur les queries structurelles.
. **Dogfooding** — Le plugin plantuml-gradle consomme son propre service pour générer sa documentation. Même LlmService, même ApiKeyPool, même ValidatePlantumlSyntaxTask. Zéro duplication.
. **Adaptateur unique** — `GraphifyPromptAdapter.kt` (~60 lignes) convertit un sous-graphe Graphify en fichier `.prompt`. Le pipeline existant traite ce `.prompt` exactement comme un prompt manuel.
. **Généralisation** — LlmService + ApiKeyPool + ConfigLoader + GraphifyPromptAdapter doivent migrer vers bakery-gradle pour être réutilisés par training-gradle, codebase-gradle, readme-gradle, et les applications Spring Boot.
. **Modèles cloud** — Ollama Cloud (Qwen 3.5/3.6, GLM 5.1) via endpoint `https://ollama.com/api/chat`. Pas de modèle local. Authentification par Device Key SSH.
. **Boucle sémantique** — Code → Graphify (knowledge graph) → Adaptateur → Plugin (LLM) → Diagrammes PNG → Graphify (ingestion) → Graphe enrichi. Le système s'améliore itérativement.

'''

== 📋 Leçons apprises

* Graphify envoie le contenu des fichiers docs (`.adoc`) à l'API LLM pour l'extraction sémantique. Les fichiers code sont extraits localement par tree-sitter. Il faut exclure les credentials du `.graphifyignore`.
* L'always-on hook de Graphify rappelle à l'agent de lire GRAPH_REPORT.adoc avant de faire un grep. Il n'oblige pas — l'agent reste libre de choisir.
* Le patron de génération est le même pour tous les plugins : GraphifyPromptAdapter → `.prompt` → pipeline existant. Seul le contenu du prompt change.
* La Phase 1 (dogfooding) ne nécessite que 2 nouveaux fichiers Kotlin (~100 lignes total) et 1 ligne dans PlantumlManager. Aucune modification du code existant.
