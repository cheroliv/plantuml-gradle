**Document de référence** — Stratégie de gestion du contexte et de l'agent OpenCode

Ce document formalise la méthodologie LAZY/EAGER pour optimiser l'efficacité d'OpenCode
sur des projets complexes (EPIC multi-sessions, 10+ fichiers, 100+ décisions).

**Inspiré de** : `plantuml-api/AGENT_MODUS_OPERANDI.adoc` — Adapté au projet `plantuml-plugin`
--

[WARNING]
====
**RACINE DE TRAVAIL** : Ce fichier est dans `plantuml-plugin/`, qui est la **racine de travail** du projet.
La racine du dépôt git est `plantuml-gradle/`, mais l'agent travaille depuis `plantuml-plugin/`.
TOUS les chemins dans ce document sont relatifs à `plantuml-plugin/`.
====

== 🎯 PARTIE 1 — VUE D'ENSEMBLE

=== 1.1 Problème Résolu

**Contexte** : Les LLM (OpenCode, etc.) ont une fenêtre de contexte limitée (~100k-200k tokens).

**Symptômes** :

* ❌ Saturation du contexte (trop de fichiers chargés)
* ❌ Dilution de l'attention (LLM perdu dans les détails)
* ❌ Tokens gaspillés sur des infos non critiques
* ❌ Hallucinations (LLM invente car contexte saturé)
* ❌ Rappels chronophages (réexpliquer à chaque session)

**Solution** : Séparer l'information en 2 catégories (**LAZY** / **EAGER**)

=== 1.2 Principes Fondamentaux

[cols="1,4", options="header"]
|===
|Principe |Description

|**LAZY/EAGER**
|Séparer l'information critique (toujours chargée) des détails (chargés sur demande).
**EAGER** = Tableau de bord de voiture. **LAZY** = Manuel du propriétaire.

|**1 session = 1 objectif**
|Une session traite UNE seule tâche (1 fichier, 1 feature, 1 bug).
Durée cible : 15-30 minutes.

|**Archive systématique**
|Chaque session terminée est archivée avec résumé et prochaines étapes.
Rien ne se perd, tout se retrouve.

|**Git manuel**
|L'agent NE JAMAIS committer sans permission explicite.
C'est l'utilisateur qui gère Git.

|**Tests sur demande**
|L'agent NE JAMAIS lancer de tests en fin de session sans demande.
Les tests prennent 1-3 minutes.
|===

=== 1.3 Analogie

[cols="1,2,2", options="header"]
|===
|Type |EAGER (toujours chargé) |LAZY (sur demande)

|**Voiture**
|Tableau de bord (vitesse, carburant, alertes)
|Manuel du propriétaire (détails techniques, historique)

|**Projet**
|`PROMPT_REPRISE.adoc` + `*_ESSENTIALS.adoc`
|Archives sessions, références détaillées

|**Taille**
|≤ 100 lignes total (~5k tokens)
|Illimité (lu uniquement si besoin)

|**Chargement**
|Automatique (début session)
|Sur demande explicite ("Je charge X ?")
|===

=== 2.3 Comment l'Agent Sait Quoi Charger ?

==== Début de Session

**Chargement automatique (EAGER)** :
1. `PROMPT_REPRISE.adoc` → Mission, priorités, contexte
2. `.agents/INDEX.adoc` (section règles) → Règles absolues
3. `*_ESSENTIALS.adoc` (si EPIC active) → Contexte métier critique

**Si besoin de détails** :
* Architecture ? → `.agents/API_KEY_POOL_REFERENCE.adoc` (LAZY)
* Détails session N ? → `.agents/sessions/{N}-{titre}.adoc` (LAZY)
* Procédure fin ? → `.agents/PROCEDURES.adoc` (LAZY)

==== Fin de Session

**Mots-clés déclencheurs** :
* `"nouvelle session"`
* `"je quitte"`
* `"session terminée"`
* `"à plus tard"`
* `"on arrête là"`

**Action** : Exécuter les 5 étapes de la procédure d'archivage (Section 4.3) **SILENCIEUSEMENT**

== ⚠️ PARTIE 3 — RÈGLES ABSOLUES

=== 3.1 Commits/Git — INTERDICTION FORMELLE

**L'agent NE DOIT JAMAIS** exécuter de commit, push, merge, ou toute commande git
modifiant l'historique **SANS permission explicite de l'utilisateur**.

[cols="1,4", options="header"]
|===
|Autorisé |`git status`, `git diff`, `git log`, `git show` (lecture seule)

|Interdit
|`git add`, `git commit`, `git push`, `git merge`, `git rebase` (sauf ordre explicite)
|===

**Procédure obligatoire avant tout commit** :

1. Montrer les modifications (`git diff --stat`)
2. Demander : `"Veux-tu que je commit ces changements ?"`
3. **Attendre confirmation explicite** (`"oui"`, `"commit"`, `"vas-y"`)
4. **Seulement après** : exécuter le commit

=== 3.2 Tests en Fin de Session — INTERDICTION FORMELLE

**L'agent NE DOIT JAMAIS** lancer de tests (`./gradlew test`, etc.) lors de la
procédure de fin de session **SANS permission explicite de l'utilisateur**.

**Raison** : Les tests prennent 1-3 minutes et ralentissent inutilement la clôture.

=== 3.3 Secrets et Credentials — SÉCURITÉ MAXIMALE

**INTERDICTION ABSOLUE** :

* ❌ Jamais de secrets en dur dans le code
* ❌ Jamais de credentials dans les fichiers de config
* ❌ Jamais de tokens API commités dans Git

**Solution** : Utiliser des références (`keyRef: ${GOOGLE_API_KEY_1}`)

== 🔄 PARTIE 4 — CYCLE DE VIE D'UNE SESSION

=== 4.1 Démarrage de Session

==== Prompt d'Ouverture (Template)

[source]
----
Nouvelle session. Contexte chargé :
1. PROMPT_REPRISE.adoc (mission en cours)
2. INDEX.adoc (règles absolues)
3. API_KEY_POOL_ESSENTIALS.adoc (contexte métier)

Mission : [Décrire la tâche UNIQUE de cette session]

Contrainte : 1 fichier à la fois, validation après chaque changement.
----

=== 4.2 Pendant la Session

==== Règles de Travail

* ✅ **1 fichier à la fois** : Modifier un seul fichier avant de passer au suivant
* ✅ **Validation immédiate** : Compiler après chaque modification
* ✅ **Contexte léger** : Maximum 50k tokens EAGER (~3000 lignes)
* ✅ **Progression visible** : Mettre à jour `PROMPT_REPRISE.adoc` en temps réel

=== 4.3 Fin de Session — Procédure en 5 Étapes OBLIGATOIRES

[IMPORTANT]
====
**RÈGLE ABSOLUE** : Quand un déclencheur de fin de session est détecté, l'agent DOIT exécuter **TOUTES les 5 étapes** **SANS EXCEPTION et SANS OMISSION**.

L'agent **NE DOIT JAMAIS** :
* ❌ S'arrêter avant d'avoir complété les 5 étapes
* ❌ Omettre une étape ("je ferai ça plus tard")
* ❌ Demander "est-ce que je dois faire l'étape X ?" — la réponse est **TOUJOURS oui**
* ❌ Considérer la procédure comme facultative
* ❌ Répondre "✅ Procédure exécutée" sans avoir VÉRIFIÉ chaque fichier

**L'agent DOIT** :
* ✅ Exécuter chaque étape une par une
* ✅ Vérifier que chaque fichier a été effectivement modifié
* ✅ Afficher la checklist de confirmation complète à la fin
====

==== ÉTAPE 1 : Créer l'Archive LAZY

**Fichier** : `.agents/sessions/{N}-{type}-{sujet}.adoc`

**Structure** :
```markdown
# Session {N} — {Titre}

**Date** : {date} | **Statut** : ✅/⚠️/❌ | **Type** : {feature/debug/test}

## Contexte
{1-2 paragraphes}

## Actions
| Fichier | Modification |
|---------|--------------|
| `file.kt` | -X lignes |

## Commandes
{cmd}

## Résultats
| Critère | Statut |
|---------|--------|
| Compilation | ✅ |

## Prochaines Étapes
1. {action}
```

==== ÉTAPE 2 : Mettre à jour le Résumé EAGER

**Fichier** : `PROMPT_REPRISE.adoc`

**Contraintes** :
* Maximum **70 lignes**
* **1 référence** à l'archive LAZY
* **Commandes** session N+1
* **Critères d'acceptation** clairs

==== ÉTAPE 3 : Mettre à jour SESSIONS_HISTORY.adoc

**Fichier** : `.agents/SESSIONS_HISTORY.adoc`

**Contenu** : Ajouter la ligne pour la session terminée en tête de liste

==== ÉTAPE 4 : Nettoyer INDEX.adoc

**Vérifier** :
* Date de dernière mise à jour
* Session en cours → N+1
* Session terminée → N
* Ajouter la session N dans le tableau des sessions récentes
* Mettre à jour la roadmap si nécessaire
* Mettre à jour les fichiers associés si nécessaire

==== ÉTAPE 5 : Mettre à jour TEST_COVERAGE_ANALYSIS.adoc (si applicable)

**Fichier** : `.agents/archives/tests_analysis/TEST_COVERAGE_ANALYSIS.adoc`

**Conditions** :
* ✅ **OBLIGATOIRE** si des tests ont été ajoutés, modifiés ou supprimés pendant la session
* ✅ **OBLIGATOIRE** si le nombre total de tests a changé
* ⏭️ Peut être ignoré si aucun test n'a été touché

==== ÉTAPE 6 : Mettre à jour COMPLETED_TASKS_ARCHIVE.adoc

**Fichier** : `.agents/archives/COMPLETED_TASKS_ARCHIVE_{YYYY-MM}.adoc`

**Contenu** : Ajouter un bloc pour la session terminée contenant :
* Contexte (session précédente, objectif)
* Tâches réalisées (liste avec ✅)
* Résultats (métriques, fichiers, tests)
* Leçons apprises
* Prochaine session

**Cette étape est TOUJOURS OBLIGATOIRE**, même si la session n'a produit que de la documentation.

==== Checklist de Confirmation Finale OBLIGATOIRE

**L'agent DOIT afficher cette checklist APRÈS avoir exécuté les 6 étapes** :

```
✅ Procédure de fin de session exécutée
📋 Checklist :
[✅] 1. Archive session N créée
[✅] 2. PROMPT_REPRISE.adoc mis à jour pour session N+1
[✅] 3. SESSIONS_HISTORY.adoc mis à jour
[✅] 4. INDEX.adoc mis à jour
[✅] 5. TEST_COVERAGE_ANALYSIS.adoc mis à jour (si applicable)
[✅] 6. COMPLETED_TASKS_ARCHIVE.adoc mis à jour
```

**AUCUNE étape ne peut être marquée [✅] si le fichier n'a pas été effectivement modifié et vérifié.**

**L'agent NE DOIT PAS** répondre "✅ Procédure exécutée" sans avoir vérifié chaque fichier.

=== 5.2 Signes Qu'il Faut Changer de Session

* [ ] Le LLM répète des erreurs déjà corrigées
* [ ] Plus de 3 fichiers modifiés en parallèle
* [ ] Conversation > 50 messages
* [ ] Tu te dis `"il faudrait qu'on reparte sur de bonnes bases"`

**Règle d'or** : Mieux vaut 5 sessions de 20 minutes qu'une session de 2 heures. 🎯

== 🎯 PARTIE 6 — TYPES DE SESSIONS ET DÉTECTION

=== 6.1 Tableau de Détection

[cols="1,2,2,2", options="header"]
|===
|# |Type de Session |Indices Détectés |Méthodologie Proposée

|1
|**Création fichier**
|`"créer"`, `"models.kt"`, `"data class"`
|`"Veux-tu que j'applique la méthodologie de création ?"`

|2
|**Debug**
|`"debug"`, `"échec"`, `"erreur"`, `"NullPointerException"`
|Aucune — Agir directement

|3
|**Test unitaire**
|`"créer test"`, `"couverture"`, `"*Test.kt"`
|`"Veux-tu que je consulte l'analyse de couverture ?"`

|4
|**Fin de session**
|`"session terminée"`, `"je quitte"`, `"à plus tard"`
|**PROCÉDURE AUTOMATIQUE** (5 étapes)
|===

=== 6.2 Format de Proposition Obligatoire

**Template** :
```
🎯 Méthodologie détectée : [Type de session]

Je peux appliquer :
- [NOM_DU_FICHIER].adoc
- Checklist : [Point 1], [Point 2], [Point 3]

Veux-tu que je charge ce fichier et applique cette méthodologie ?
```

=== 6.3 Exceptions — Ne PAS Proposer

[cols="2,3,2", options="header"]
|===
|Cas |Raison |Comportement

|Session déjà en cours
|Méthodologie déjà chargée
|Continuer sans re-proposer

|Utilisateur a spécifié la méthode
|Déjà dans le prompt
|Charger directement (pas de question)

|Session urgente
|`"corriger bug critique"`
|Agir immédiatement

|Fin de session
|`"je quitte"`
|**PROCÉDURE AUTOMATIQUE** (pas de question)
|===

=== 7.2 Revue de Session (Hebdomadaire)

**Questions à se poser** :

1. ✅ Les sessions étaient-elles assez courtes (≤ 30 min) ?
2. ✅ Les archives sont-elles bien organisées ?
3. ✅ Le contexte est-il resté léger (< 50k tokens) ?
4. ✅ Les règles Git/tests ont-elles été respectées ?

**Actions correctives** :

* Si sessions trop longues → Découper en sous-tâches
* Si contexte saturé → Archiver plus agressivement
* Si règles non respectées → Rappeler en début de session

== ANNEXE A — GLOSSAIRE

**EAGER** : Information chargée automatiquement au début de chaque session (critique)

**LAZY** : Information chargée uniquement sur demande explicite (détails)

**Session** : Période de travail continue avec OpenCode (15-30 minutes idéalement)

**Archive** : Fichier Markdown détaillant une session terminée

**Contexte** : Ensemble des informations chargées dans la fenêtre de conversation OpenCode

**Token** : Unité de texte pour les LLM (~4 caractères = 1 token)

**EPIC** : Ensemble de fonctionnalités liées (ex: EPIC Pool de Clés API)
